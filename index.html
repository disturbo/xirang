S1 Overview

方法论概述

核心问题、设计目标与版本关系

V8 宪法 + V8.5 工具链
!

解决的问题

1. 多 Agent 协作缺乏统一的状态追踪和通信规范
2. 合规规则全靠"约定调用"，Agent 可跳过
3. 子任务 spawn 无运行时记录，无法追踪/回收
4. 成本无感知，超支才发现
✓

V8.5 的解法

1. 统一 Schema 的 per-Agent 状态文件 + append-only 事件流
2. gate-enforce.py 硬门禁：P0 违规 = 操作被阻断
3. subtask-record.py CRUD + v8_spawn/v8_collect 自动维护
4. cost-fuse 熔断 + spawn-budget-check 预算评估
?

通俗类比：息壤到底是什么？

想象你管一个 5 人施工队，每人都很能干但互相看不见对方在干嘛：

没有息壤时 — 两个人可能同时改同一面墙；有人超预算了才发现；有人忘记报完工。

有息壤后 —
· 看板 = 工地白板（谁在干什么，一眼看清）
· 握手 = 开工前喊一声"我要动 3 楼东墙"（防撞车）
· 门禁 = 安全帽检查（没戴帽不让进工地，exit 1 阻断）
· 预算 = 开工前算材料费（超了就降级用便宜材料）
· 收工 = 完工拍照 + 报给监理（产物清单 + 验收方）

版本关系

V8（正文，已冻结 2026-05-19） → 设计原理 / 宪法 / 10 态状态机 / 5 层通信
V8.5（运行时扩展，2026-05-24~25） → 可执行工具链 / 门禁 / 消息队列 / 成本调度

V8.5 不替换 V8。两者共存：V8 为设计参考，V8.5 为执行标准。

核心Bashv1.4.0

v8-handshake.sh

运行时握手工具。source 后调用 4 个函数管理任务全生命周期。v1.4.0 内嵌 gate-enforce 硬门禁。

CLI 接口

  • v8_handshake <档位> <任务名> <写入范围> [验收方] [agent_id]
  • v8_spawn <task_id> <sub_id> <agent> <model> <type> <name> <scope>
  • v8_collect <task_id> <sub_id> [agent] [tokens] [cost]
  • v8_end <task_id> [agent_id] [result]
  • v8_log <summary> [result] [path] (M3 轻量)

关键行为

  • M4/M5 自动触发 gate pre-start 门禁
  • 更新 Agent 状态文件（status/current_task/heartbeat）
  • 追加 task_start / task_end 到智能体事件.jsonl
  • M3 不触发门禁（轻量设计）
  • gate-enforce.py 不存在时优雅降级（[GATE-SKIP]）
Phase 4Pythonv1.0.0

gate-enforce.py

统一门禁入口。4 个子命令覆盖任务生命周期关键点。P0 = exit 1 硬阻断。

子命令

  • pre-start — Agent 已 busy / 任务被阻塞 → block
  • pre-spawn — 无 task_id / 成本熔断 / 写入越权 → block
  • pre-write — 路径越权 / 禁止目录 → block
  • pre-end — 状态机非法转换 → block

优先级分级

  • P0: 硬阻断（exit 1）— AGENT_BUSY / FUSE_BLOWN / SCOPE_VIOLATION 等
  • P1: Advisory — BUDGET_WARNING / FRONTMATTER_MISSING / UNCOLLECTED_SUBTASKS
  • P2: Advisory — EMOJI_DETECTED / MODEL_DOWNGRADE
  • P3: Advisory — BRAND_COLOR
Phase 1Pythonv1.0.0

subtask-record.py

子任务运行时记录 CRUD。每个 spawn 的子 Agent 都有一份 JSON 记录。

子命令

  • create — 创建子任务记录（JSON 文件）
  • transition — 推进状态（PENDING → RUNNING → SUCCESS/FAILED）
  • collect — 收集结果，标记 COLLECTED
  • destroy — 销毁记录（清理）

存储位置

  • _temp/{task_id}/subtasks/{sub_id}.json
  • v8_spawn 自动调用 create，v8_collect 自动调用 collect
Phase 2Pythonv1.0.0

msg-queue.py

Agent 间结构化通知。本地 JSONL + fcntl.flock 全局锁，解决并发安全。

子命令

  • enqueue --to <agent> --type <type> --payload '{}'
  • dequeue --agent <id> — 取出最早未处理消息
  • peek --agent <id> — 只看不取
  • ack --id <msg_id> — 确认处理完毕
  • expire --max-age 3600 — 清理过期消息
Phase 3Pythonv1.0.0

spawn-budget-check.py

Spawn 前预算评估。绿灯/黄灯/红灯 + 模型降级路径。

信号灯

  • 绿灯（exit 0）：预算充裕，可 spawn
  • 黄灯（exit 1）：建议降级模型（opus→sonnet→haiku）
  • 红灯（exit 2）：所有模型均超预算，禁止 spawn

降级路径

  • opus (21.84 CNY/1K) → sonnet (4.56) → haiku (0.375) → deepseek (0.55)
  • 保留 20% buffer 作为安全边际

S7 Design Decisions

关键设计决策

V8.5 升级过程中的技术选型与架构决策

1

门禁实现方式

统一入口 + handshake 硬嵌入
gate-enforce.py 作为编排层封装既有检查，v8-handshake 在生命周期函数中硬调用，Agent 不可绕过。
2

拦截分级

P0 硬阻断 / P1-P3 Advisory
P0 违规严重到必须修复。P1-P3 输出到 stderr 提醒但不阻断，避免过度治理影响效率。
3

消息队列并发模型

全局 flock 文件锁
JSONL 文件驱动，fcntl.flock 全局锁保证 read-modify-write 原子性。10 进程并发压测零丢失。
4

M3 是否加门禁

不加
M3 是 10-20 分钟轻量操作，加门禁开销不合理。仅 M4/M5 触发 gate check。
5

工具不存在时的行为

优雅降级（不阻断）
gate-enforce.py 删除后 handshake 输出 [GATE-SKIP] warning 继续运行。回滚只需删文件。
6

V8 与 V8.5 的关系

扩展层，非替换
V8 正文永久冻结。V8.5 是可执行运行时层，两者共存。封版后只接受 bugfix。
