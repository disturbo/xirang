V9.4.3

自证回归、九源治理与运行时消费链

V9.4.3 runtime recovery：第一反射器聚合九类治理源，并在独立 runtime checks 中验证 GBrain、LLM Wiki、熵治理、skill 解析等消费者与新鲜度；统一输出到 ~/.xirang/v9-runtime/巡检/health-latest.json。成本能力已于 2026-07-19 退役。

健康判定必须同时满足 summary.active=0、sources_failed=[]、runtime checks 无 failed/stale，以及 Harness trust/hash 新鲜度验证通过。任一消费者失败都不得被治理源绿色覆盖。

S0

什么是息壤

一句话：AI Agent 团队的作战手册。定义了谁做什么、怎么流转、出错了谁修、怎么证明活干完了。

不是理论框架，是操作规程——每个 Agent 启动时读这一份，就知道自己在团队里的位置和行为边界。

V9 核心突破：将合规判定从 V8 的"六档分类"简化为一个二元判断——写文件还是不写文件。写文件就声明，不写就直接回复。所有复杂规则都是这条铁律的操作化展开。

息壤的核心能力

维度息壤做了什么
角色分工6 个原子 Agent 各有职责边界，不越权
流程管道六步流程：拆分→计划→执行→监控→检核→验收
通信协议CLI 直连（同步）+ Vault 文件（持久化审计）双层架构
合规框架二元触发器 + M3/M4/M5 声明制度 + Gate 门禁自动拦截
错误处理五级升级链 + 回滚补偿；Phoenix 当前仅为设计参考
成本治理已退役；历史设计保留，不参与当前运行
质量门禁pre-write-hook + gate-enforce + 产物自检三件套
S0.1

「息壤」命名释义

「息壤」出自《山海经·海内经》鲧治水神话。息壤是能自行增生、取之不竭的神土。核心意象是会自己生长的基础材料。

《山海经》息壤意象方法论对应
自行生长的土task card、retrospective、规则、脚本在实践中持续增殖
用来应对洪水应对多 Agent 协作中的上下文洪水、任务洪水、漂移洪水
不是普通材料，而是基础设施不是单篇文档，而是入口、门禁、状态表、事件流、度量系统
鲧的失败：有神土也不能无治理地堵不能只堆文档，必须有运行时激活、路由、升级和复盘
后续治水走向疏导与定九州目标不是塞进一个 Agent，而是拆分、分流、验收、沉淀

「息壤」不是「无限扩张」的隐喻，而是「可生长且需治理」的隐喻。

S0.2

前提：项目记忆系统

息壤的前提不是必须拥有 Obsidian Vault，而是必须有一个可持久化的项目记忆系统。

能力为什么需要
可持久化任务、资料、决策、产物不能只留在聊天记录里
可检索Agent / 人接手时能重新找到上下文
可链接需求、资料、方案、PRD、决策、复盘能互相引用
可审计谁基于什么资料做了什么判断，有记录可追溯
可演化模板、规范、案例、门禁能随着实践持续生长

形态适用对象载体

S1

V9 核心：二元触发器与合规框架

设计动机

V8 合规层的核心问题：

  1. M0-M5 六档判定过重，Agent 认知负担大
  2. 格式不统一，6 Agent 跨 4 平台各自维护
  3. 验证困难，无法程序化检测

二元触发器

收到消息 → 第一判断：「这个任务会写文件吗？」

  • 不写 → 直接回复（零合规开销）
  • 写 → pre-flight（声明 + 亮灯 + 执行）

唯一铁律：写了文件但没声明 = 违规。

灰区判定

信号判定
有具体对象 + 明确产出物写文件 → pre-flight
咨询语气（"怎么推进"/"能不能做"）不写文件 → 直接回复
无法判定问用户确认，1 轮内决定
S2

档位系统（M3/M4/M5）

档位场景文件数时间声明格式
M3单文件修改，<20min1<20min一行声明
M4多文件产出/交付物2+20min-2h完整 V9 已激活 块
M5跨 Agent / >2h / 需拆分多>2hM4 + 预算 + 拆分计划

判定决策树

  • 只改 1 个文件 + <20min → M3
  • 改多文件 / 有明确交付物 → M4
  • 跨 Agent / >2h / 需拆分 → M5

升降档规则

  • M3 发现需写第 2 个文件 → 升为 M4
  • M4 发现需跨 Agent 协作 → 升为 M5
  • M4/M5 已亮灯 → 不可降档

六脉体系：六个子系统

脉子系统一句话负责人

脉① Claude Design 工艺

方案设计和 PRD/原型大块产出的工程方法。核心理念：价值在逼近过程，而非一次性产物。

3 Phase x 10 Step 工作流

  • Phase 1: 构建 Design System（资料→画像→Review）
  • Phase 2: 需求收敛 → 探索对齐
  • Phase 3: 产物生成 → 实现

脉② 驾驭工程

Agent 的可靠性、可观测性、治理。三层可观测性：状态层、事件层、指标层。现行门禁保留 3 个检查点：pre-start、pre-write、handoff；cost-fuse 已退役。

脉③ 不死鸟 Phoenix

design/reference，未部署。 8 个抗体模式与 10 步排查流程是候选设计；当前错误在有限重试后直接升级 Owner，不声称自动修复已发生。

脉④ LLM Wiki

三层资料结构：Source（原始源）→ Summary（摘要层）→ Published（发布层）。

脉⑤ GBrain

语义搜索引擎：bge-m3 嵌入 + PGLite 存储 + 混合搜索 + dream/sync 维护。

脉⑥ Skill 清单

各平台 skill 目录 + 能力声明 + 发现机制。SKILL.md 标准格式：When / Check / Steps / Chain / Quality Bar。

S9.5

外部 Agent 接入协议

四条最低要求

  1. CLAUDE.md 前置声明 — 外部 Agent 必须声明身份和意图
  2. 路径白名单 — 只能写入指定暂存路径
  3. 三检义务 — frontmatter / 路径合规 / 无越权
  4. 事后补办 — 漏检可补，但不能跳过

信任边界

外部 Agent 只能写暂存区，不可自行验收或移动到正式路径。

S11

Gate 与 Hook 强制机制

gate-enforce.py 规则（P0硬阻断）

规则说明
AGENT_BUSYAgent 状态为 busy 时拒绝新任务
TASK_BLOCKED任务被依赖阻塞时不可启动
NO_TASK_ID写操作无关联任务卡时阻断
FUSE_BLOWN历史规则；成本治理退役后不再参与现行门禁
SCOPE_EXCEEDED写入路径超出声明范围
PATH_FORBIDDEN写入禁止路径（如其他Agent专属路径）
MISSING_DELIVERABLES标记完成但缺少必需产物

Hook 层级

  • Layer 0：不可篡改（系统级保护）
  • Layer 1A：禁止路径（路径白名单/黑名单）
  • Layer 1B：任务卡授权（基于 task card 的动态授权）
