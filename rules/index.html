运行时契约

8 条铁律

每个 Agent 启动时必须加载的核心约束。违反任何一条即判定违规。

铁律 1：二元触发器

收到消息 → 第一判断是「会写文件吗」→ 写 = 声明，不写 = 直接回复。没有第三种情况。

铁律 2：违规定义

写了文件但没声明 = 违规。这是唯一的铁律。所有复杂规则都是这条的操作化展开。

铁律 3：声明格式
  • M3: 一行 V9 写入声明：{路径} | {摘要}
  • M4: 完整 V9 已激活： 块（档位 + 任务 + 范围 + 验收方）
  • M5: M4 + 预算 + 拆分计划
铁律 4：豁免清单
  • 追加运行日志一行（M3 收工）免声明
  • heartbeat 更新（状态文件 last_heartbeat 字段）免声明
  • 仅此两项，无其他豁免
铁律 5：灰区处理
  • 有具体对象 + 明确产出物 → 写文件
  • 咨询语气 → 不写
  • 无法判定 → 问用户，1轮内确认
铁律 6：不写文件场景

解释 / 讨论 / 评审 / 闲聊 / 巡检 = 不写文件 = 零合规开销。闲聊不是任务。

铁律 7：执行中扩展

发现要写更多文件 → 暂停 → 补充声明 → 再继续。每次写入前自问：「我声明过要写这个文件吗？」没有 → 先声明。

铁律 8：提交≠验收

submitted 不等于 accepted。未经验收方确认，不得声称已验收。

现行门禁保留 pre-start / pre-write / handoff；cost-fuse 已于 2026-07-19 退役。历史规则按 P0-P3 保留说明。

豁免与灰区

什么情况可以不严格遵循

日志追加（M3 收工一行）

追加运行日志一行免声明。这是最常见的豁免。格式：- HH:MM | M3 | 动作摘要 | 结果 | 产物路径(可选)

Heartbeat 更新

状态文件的 last_heartbeat 字段更新免声明。V9.2 提供 heartbeat-update.sh 和 heartbeat-check.sh 自动化脚本，三级超时检测(15min/30min/60min)。

V9 豁免路径

pre-write-hook.sh 对 .standards/event-rotate.py 和 heartbeat 更新自动放行。这是系统级白名单，非 Agent 可主动触发。

灰区：改1个文件但更新看板

看板也是文件，所以是 M4（2个文件）。需完整声明。不要因为看板"只是更新状态"就忽略它。

灰区：批量修改同一字段

即使内容一致，改 3+ 文件 = M4。档位取决于文件数量和复杂度，不取决于"改动是否一样"。

外部 Agent 接入最低要求

非息壤原生 Agent 接入协作时的最小合规集

1

写入声明

外部 Agent 写文件前也必须声明。格式同 M3/M4。无声明写入会被 gate-enforce 硬拦截。

2

路径隔离

只能写入分配的路径，不碰 .standards/ 和其他 Agent 的文件。越界写入 = P0 拦截。

3

状态报告

任务开始和结束时更新看板，或通过消息区报告。长任务需 heartbeat。

4

摘要回传

完成后返回 ≤500 字摘要给调用方，不泄漏全部上下文。摘要包含：产物清单 + 状态 + 风险。
